# Workspace Surface Audit

「このワークスペースとマシンは今実際に何ができるのか、次に何を追加・有効化すべきか？」に答えるための読み取り専用監査スキルです。

これはセットアップ監査プラグインに対する ECC ネイティブの答えです。ユーザーが明示的にフォローアップの実装を求めない限り、ファイルを変更しません。

## 使うタイミング

- ユーザーが「Claude Code をセットアップ」「自動化を推奨」「どのプラグインや MCP を使うべき？」「何が不足している？」と言う
- スキル、フック、コネクタを追加インストールする前のマシンやリポジトリの監査
- 公式マーケットプレイスプラグインと ECC ネイティブカバレッジの比較
- `.env`、`.mcp.json`、プラグイン設定、接続アプリ面をレビューして欠落ワークフロー層を発見
- 機能をスキル、フック、エージェント、MCP、外部コネクタのどれにすべきか判断

## 絶対ルール

- シークレット値を印刷しない。プロバイダー名、機能名、ファイルパス、キーまたは設定が存在するかのみ表面化。
- ECC が合理的にその面をオーナーできる場合、汎用的な「もう 1 つプラグインを入れる」アドバイスより ECC ネイティブのワークフローを優先。
- 外部プラグインはベンチマークやインスピレーションとして扱い、権威的なプロダクト境界ではない。
- 以下の 3 つを明確に分離：
  - すでに今利用可能
  - 利用可能だが ECC で上手くラップされていない
  - 利用不可で新しい統合が必要

## 監査入力

問いに上手く答えるために必要なファイルと設定のみを調査：

1. リポジトリ面
   - `package.json`、ロックファイル、言語マーカー、フレームワーク設定、`README.md`
   - `.mcp.json`、`.lsp.json`、`.claude/settings*.json`、`.codex/*`
   - `AGENTS.md`、`CLAUDE.md`、インストールマニフェスト、フック設定
2. 環境面
   - アクティブなリポジトリと明らかな隣接 ECC ワークスペースの `.env*` ファイル
   - `STRIPE_API_KEY`、`TWILIO_AUTH_TOKEN`、`FAL_KEY` などのキー名のみ表面化
3. 接続ツール面
   - インストール済プラグイン、有効なコネクタ、MCP サーバー、LSP、アプリ統合
4. ECC 面
   - 既にニーズをカバーする既存のスキル、コマンド、フック、エージェント、インストールモジュール

## 監査プロセス

### フェーズ 1：既存の棚卸し

コンパクトなインベントリを生成：

- アクティブなハーネスターゲット
- インストール済プラグインと接続アプリ
- 設定済 MCP サーバー
- 設定済 LSP サーバー
- キー名から推測される env バック型サービス
- ワークスペースに既に関連する既存 ECC スキル

面がプリミティブとしてのみ存在する場合、それを明示してください。例：

- 「Stripe は接続アプリ経由で利用可能だが、ECC には billing-operator スキルがない」
- 「Google Drive は接続済みだが、ECC ネイティブの Google Workspace オペレーターワークフローが存在しない」

### フェーズ 2：公式・インストール済面に対するベンチマーク

以下に対してワークスペースを比較：

- セットアップ、レビュー、ドキュメント、デザイン、ワークフロー品質で重なる公式 Claude プラグイン
- Claude または Codex にローカルインストールされたプラグイン
- ユーザーが現在接続しているアプリ面

ただ名前を列挙しない。各比較で以下に答える：

1. 実際に何をするか
2. ECC が既にパリティを持つか
3. ECC がプリミティブのみか
4. ECC にワークフロー自体が欠落しているか

### フェーズ 3：ギャップを ECC 決定に変換

全ての実在のギャップについて、正しい ECC ネイティブ形状を推奨：

| ギャップタイプ | 推奨 ECC 形状 |
|----------|---------------------|
| 反復可能なオペレーターワークフロー | スキル |
| 自動強制または副作用 | フック |
| 専門化された委譲ロール | エージェント |
| 外部ツール橋渡し | MCP サーバーまたはコネクタ |
| インストール/ブートストラップガイダンス | セットアップまたは監査スキル |

ニーズがインフラではなく運用である場合、既存ツールをオーケストレートするユーザー向けスキルをデフォルトにしてください。

## 出力フォーマット

以下の順序で 5 つのセクションを返します：

1. **現在の面**
   - 今すぐ使えるもの
2. **パリティ**
   - ECC がベンチマークに一致または超えている箇所
3. **プリミティブのみのギャップ**
   - ツールは存在するが ECC にクリーンなオペレータースキルがない
4. **欠落統合**
   - まだ利用不可能な機能
5. **次の動きトップ 3〜5**
   - インパクト順に並べた具体的な ECC ネイティブ追加案

## 推奨ルール

- カテゴリごとに最高価値のアイデアを 1〜2 個に絞る。
- 明白なユーザー意図とビジネス価値を持つスキルを優先：
  - セットアップ監査
  - 課金/顧客オペレーション
  - Issue/プログラムオペレーション
  - Google Workspace オペレーション
  - デプロイ/オペレーション制御
- コネクタが企業固有の場合、真に利用可能か、ユーザーのワークフローに明確に有用な場合のみ推奨。
- ECC が既に強いプリミティブを持つなら、新サブシステムを発明するよりラッパースキルを提案。

## 良い成果

- ユーザーが接続済み、欠落、ECC が次にオーナーすべきものを即座に見える。
- 推奨が、再度の探索パスなしにリポジトリで実装できるほど具体的。
- 最終的な答えが API ブランドではなくワークフローを中心に整理されている。
