# Project Flow Ops

このスキルは、切り離された GitHub Issue、PR、Linear タスクを 1 つの実行フローに統合します。

問題がコードではなく調整である場合に使用してください。

## 使うタイミング

- オープンな PR や Issue のバックログをトリアージ
- Linear に載せるべきものと GitHub のみに留めるべきものを判断
- アクティブな GitHub 作業を内部実行レーンにリンク
- PR を merge、port/rebuild、close、park に分類
- レビューコメント、CI 失敗、古い Issue が実行をブロックしているか監査

## 運用モデル

- **GitHub** は公開・コミュニティの真実
- **Linear** はアクティブに予定された作業の内部実行の真実
- 全ての GitHub Issue に Linear Issue が必要なわけではない
- Linear の作成・更新は以下の場合のみ：
  - アクティブ
  - 委任済み
  - スケジュール済み
  - クロスファンクショナル
  - 内部追跡に値するほど重要

## 中核ワークフロー

### 1. まず公開面を読む

以下を収集：

- GitHub Issue または PR の状態
- 作者とブランチの状態
- レビューコメント
- CI ステータス
- リンクされた Issue

### 2. 作業を分類

各項目は以下のいずれかの状態に落ち着く：

| 状態 | 意味 |
|-------|---------|
| Merge | 自己完結し、ポリシー準拠、準備完了 |
| Port/Rebuild | 良いアイデアだが ECC 内で手動で再着地させるべき |
| Close | 方向が違う、古い、安全でない、または重複 |
| Park | 潜在的に有用だが、今はスケジュールされていない |

### 3. Linear が必要かを判断

以下の場合のみ Linear を作成・更新：

- 実行がアクティブに計画されている
- 複数のリポジトリまたはワークストリームが絡む
- 作業に内部オーナーシップやシーケンシングが必要
- Issue がより大きなプログラムレーンの一部

機械的に全てをミラーしないでください。

### 4. 2 つのシステムを整合させる

作業がアクティブなとき：

- GitHub の Issue/PR は公開面で何が起きているかを示す
- Linear はオーナー、優先度、実行レーンを内部で追跡

作業がリリースまたは却下されたとき：

- 公開面の解決を GitHub に戻す
- Linear タスクを適切にマークする

## レビュールール

- タイトル、サマリ、信頼のみで merge しない。差分全体を使う
- 外部ソース機能が価値あるが自己完結していない場合は ECC 内で再構築する
- CI が赤なら分類して修正またはブロックする。merge 可能を装わない
- 本当のブロッカーがプロダクトの方向性なら、ツールの影に隠れずそう言う

## 出力フォーマット

以下を返します：

```text
PUBLIC STATUS
- Issue / PR の状態
- CI / レビュー状態

CLASSIFICATION
- merge / port-rebuild / close / park
- 1 段落の理由

LINEAR ACTION
- 作成 / 更新 / Linear 項目不要
- 該当する場合はプロジェクト / レーン

NEXT OPERATOR ACTION
- 正確な次の動き
```

## 良いユースケース

- 「オープンな PR バックログを監査して、merge と rebuild の判断を教えて」
- 「GitHub Issue を ECC 1.x と ECC 2.0 のプログラムレーンにマッピングして」
- 「これに Linear Issue が必要か、GitHub のみに留めるべきかチェックして」
