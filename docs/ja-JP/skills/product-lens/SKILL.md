# Product Lens — 構築前に考える

このレーンは、実装可能な仕様書ではなくプロダクト診断を担います。

ユーザーが恒常的な PRD-to-SRS や能力契約のアーティファクトを必要とする場合は、`product-capability` に引き継いでください。

## 使うタイミング

- 機能に着手する前 — 「なぜ」を検証する
- 週次のプロダクトレビュー — 本当に作るべきものを作っているか？
- 機能の選択に迷ったとき
- ローンチ前 — ユーザージャーニーの妥当性を確認
- 曖昧なアイデアを、エンジニアリング計画を始める前のプロダクトブリーフに変換するとき

## 仕組み

### モード 1：プロダクト診断

YC のオフィスアワーのように、ただし自動化。厳しい質問を投げかけます：

```
1. これは誰のためのもの？（「開発者」ではなく、具体的な人物）
2. 何が痛みか？（定量化：頻度、深刻度、現状の回避策は？）
3. なぜ今？（これを可能/必要にした変化は何か？）
4. 10 スター版はどんな姿？（お金と時間が無制限なら）
5. MVP は？（テーゼを証明する最小限のもの）
6. アンチゴールは？（明示的に作らないものは？）
7. うまくいっていることをどう判断する？（雰囲気ではなく指標）
```

出力：回答、リスク、Go/No-Go 推奨を含む `PRODUCT-BRIEF.md`。

結果が「はい、作る」なら、次のレーンは `product-capability` であって、さらなる創業者劇場ではありません。

### モード 2：創業者レビュー

現在のプロジェクトを創業者レンズでレビューします：

```
1. README、CLAUDE.md、package.json、最近のコミットを読む
2. 推測：これは何になろうとしているのか？
3. スコア：プロダクトマーケットフィットのシグナル（0〜10）
   - 利用量の成長曲線
   - リテンション指標（リピート貢献者、再訪ユーザー）
   - 売上シグナル（価格ページ、課金コード、Stripe 連携）
   - 競合防御力（真似されにくい要素は何か？）
4. 特定：これを 10 倍にする唯一のこと
5. フラグ：意味のないものを作っていないか
```

### モード 3：ユーザージャーニー監査

実際のユーザー体験をマッピングします：

```
1. 新規ユーザーとしてプロダクトをクローン/インストール
2. 摩擦点を全て記録（混乱する手順、エラー、不足したドキュメント）
3. 各ステップの時間を計測
4. 競合のオンボーディングと比較
5. スコア：価値実現までの時間（最初の成功まで何分？）
6. 推奨：オンボーディングの修正トップ 3
```

### モード 4：機能優先度付け

10 個のアイデアから 2 つを選ぶとき：

```
1. 候補機能を全てリストアップ
2. それぞれをスコア：インパクト（1〜5）× 確信度（1〜5）÷ 工数（1〜5）
3. ICE スコアでランク付け
4. 制約を適用：ランウェイ、チームサイズ、依存関係
5. 出力：根拠付きの優先度付けされたロードマップ
```

## 出力

全モードがエッセイではなく実行可能なドキュメントを出力します。全ての推奨には具体的な次のステップがあります。

## 連携

以下と組み合わせます：
- `/browser-qa` — ユーザージャーニー監査の発見事項を検証
- `/design-system audit` — ビジュアル磨き込みの評価
- `/canary-watch` — ローンチ後のモニタリング
- `product-capability` — プロダクトブリーフを実装可能な能力プランに昇格するとき
