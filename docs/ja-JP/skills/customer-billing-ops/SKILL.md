# Customer Billing Ops

このスキルは、汎用的な決済 API 設計ではなく、実際の顧客オペレーションに使用してください。

目的はオペレーターが次の問いに答えることを助けることです：この顧客は誰で、何が起きたか、最も安全な修正は何か、どんなフォローアップを送るべきか。

## 使うタイミング

- 顧客が課金が壊れていると言う、返金を求める、解約できない
- 重複サブスクリプション、誤請求、更新失敗、チャーンリスクの調査
- プランミックス、アクティブサブスクリプション、年額 vs 月額の転換、チームシートの混乱のレビュー
- 請求ポータルフローの作成または検証
- サブスクリプション、インボイス、返金、支払方法に関わるサポート苦情の監査

## 推奨ツール面

- 最初に Stripe などの接続された課金ツールを使用
- メール、GitHub、Issue トラッカーは補助証拠としてのみ使用
- プラットフォームが既に必要な制御を提供している場合、カスタムアカウント管理コードよりホスト型課金/顧客ポータルを優先

## ガードレール

- 応答にシークレットキー、カード詳細、不要な顧客 PII を露出しない
- 盲目的に返金しない。まず問題を分類する
- 以下を区別：
  - 誤った重複購入
  - 意図的なマルチシート/チーム購入
  - 壊れたプロダクト / 価値未達成
  - 失敗またはチェックアウト未完了
  - セルフサーブ制御不足による解約
- 年間プラン、チームプラン、日割り状態については、アクションを取る前に契約形状を確認

## ワークフロー

### 1. 顧客をクリーンに特定

利用可能な最強の識別子から開始：

- 顧客メール
- Stripe 顧客 ID
- サブスクリプション ID
- インボイス ID
- 課金に遡れることが分かっている GitHub ユーザー名またはサポートメール

簡潔な本人確認サマリを返します：

- 顧客
- アクティブサブスクリプション
- キャンセル済サブスクリプション
- インボイス
- 重複アクティブサブスクリプションなどの明らかな異常

### 2. 問題を分類

行動する前にケースを 1 つのバケツに入れる：

| ケース | 典型的なアクション |
|------|----------------|
| 個人の重複サブスクリプション | 余分を解約、返金を検討 |
| 実際のマルチシート/チーム意図 | シートを保持、課金モデルを明確化 |
| 支払失敗 / チェックアウト未完了 | ポータルまたは支払方法更新で復旧 |
| セルフサーブ制御不足 | ポータル、解約パス、インボイスアクセスを提供 |
| プロダクト障害または信頼崩壊 | 返金、謝罪、プロダクト課題を記録 |

### 3. 最も安全で可逆な行動を最初に

推奨順序：

1. セルフサーブ管理を復旧
2. 重複または壊れた課金状態を修正
3. 影響のあったチャージまたは重複のみを返金
4. 理由を文書化
5. 短い顧客フォローアップを送信

修正にプロダクト作業が必要な場合、以下を分離：

- 今すぐの顧客救済
- バックログ用のプロダクトバグ / ワークフローギャップ

### 4. オペレーター側のプロダクトギャップをチェック

顧客の痛みがオペレーター面の不足から来ている場合、明示的に呼び出してください。よくある例：

- 請求ポータルなし
- 使用量/レート制限の可視性なし
- プラン/シートの説明なし
- 解約フローなし
- 重複サブスクリプションガードなし

それらは単なるサポートインシデントではなく、ECC または Web サイトのフォローアップ項目として扱ってください。

### 5. オペレーター引き継ぎを生成

以下で終了：

- 顧客状態サマリ
- 実施したアクション
- 売上インパクト
- 送信するフォローアップ文面
- 作成するプロダクトまたはバックログ Issue

## 出力フォーマット

以下の構造を使用：

```text
CUSTOMER
- 名前 / メール
- 関連アカウント識別子

BILLING STATE
- アクティブサブスクリプション
- インボイスまたは更新状態
- 異常

DECISION
- 問題分類
- なぜこのアクションが正しいか

ACTION TAKEN
- 返金 / 解約 / ポータル / 無操作

FOLLOW-UP
- 短い顧客メッセージ

PRODUCT GAP
- プロダクトまたは Web サイトで修正すべきこと
```

## 良い推奨の例

- 「正しい修正はまだカスタムダッシュボードではなく、請求ポータル」
- 「これは本物のチームシート購入ではなく、個人の重複チェックアウトに見える」
- 「重複のチャージを 1 件返金し、残りのアクティブサブスクリプションを維持。必要なら後で組織課金に変換」
