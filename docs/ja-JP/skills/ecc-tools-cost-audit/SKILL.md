# ECC Tools Cost Audit

ECC Tools GitHub App がコストを燃やしている、PR を過剰作成している、利用制限をバイパスしている、または無料ユーザーをプレミアム分析経路に誘導していると疑われるときにこのスキルを使用してください。

これは姉妹 [ECC-Tools](../../ECC-Tools) リポジトリ向けの焦点を絞ったオペレーターワークフローです。汎用的な課金スキルではなく、リポジトリ全体のコードレビューパスでもありません。

## スキルスタック

関連する場合、以下の ECC ネイティブスキルをワークフローに組み込んでください：

- `autonomous-loops` — Webhook、キュー、課金、リトライをまたぐ有界の多段監査
- `agentic-engineering` — リクエストパスを離散的で証明可能な単位に分解
- `customer-billing-ops` — リポジトリの振る舞いと顧客影響の計算をクリーンに分ける必要があるとき
- `search-first` — ヘルパーを発明したりリポジトリローカルユーティリティを再実装する前に
- `security-review` — 認証、利用ゲート、エンタイトルメント、シークレットに触れるとき
- `verification-loop` — 再実行の安全性と修正後の正確な状態を証明
- `tdd-workflow` — ワーカー、ルーター、課金パスで修正に回帰カバレッジが必要なとき

## 使うタイミング

- ユーザーが ECC Tools のバーンレート、PR 再帰、過剰作成 PR、利用制限バイパス、プレミアムモデル漏れと言う
- タスクが姉妹 `ECC-Tools` リポジトリ内にあり、Webhook ハンドラ、キューワーカー、利用予約、PR 作成ロジック、有料ゲート強制に依存する
- 顧客報告でアプリが PR を作りすぎた、誤課金、または使える結果を出さずにコード分析したと言われる

## スコープガードレール

- `everything-claude-code` ではなく姉妹 `ECC-Tools` リポジトリで作業
- ユーザーが明確に修正を求めない限り読み取り専用から開始
- 分析バーンを追跡中に無関係な課金、チェックアウト、UI フローを変更しない
- アプリ生成ブランチとアプリ生成 PR は、証明されない限り赤旗の再帰経路として扱う
- 以下の 3 つを明示的に分離：
  - リポジトリ側のバーン根本原因
  - 顧客向け課金インパクト
  - バックログフォローアップが必要なプロダクトまたはエンタイトルメントギャップ

## ワークフロー

### 1. リポジトリスコープを凍結

- 姉妹 `ECC-Tools` リポジトリに切り替え
- まずブランチとローカル差分を確認
- 監査対象の正確な面を特定：
  - Webhook ルーター
  - キュープロデューサー
  - キューコンシューマー
  - PR 作成パス
  - 利用予約 / 課金パス
  - モデルルーティングパス

### 2. 理論化の前に入口をトレース

- 最初に `src/index.*` またはメインエントリポイントを調査
- 修正を提案する前に全てのエンキューパスをマップ
- どの GitHub イベントがキュータイプを共有するか確認
- push、pull_request、synchronize、comment、手動再実行のイベントが同じ高コストパスに収束するか確認

### 3. ワーカーと副作用をトレース

- 分析を処理するキューコンシューマーまたはスケジュールワーカーを調査
- キューされた分析が常に以下で終わるか確認：
  - PR 作成
  - ブランチ作成
  - ファイル更新
  - プレミアムモデル呼び出し
  - 利用量増加
- 分析がトークンを消費して出力が永続化される前に失敗し得るなら、壊れた出力を伴うバーンと分類

### 4. 高シグナルのバーンパスを監査

#### PR 重複

- PR ヘルパーとブランチ命名を調査
- 重複排除、synchronize イベント処理、既存 PR 再利用をチェック
- アプリ生成ブランチが分析に再突入できるなら、優先度 0 の再帰リスクとして扱う

#### クォータバイパス

- クォータがチェックされる場所と利用が予約または増加される場所を調査
- クォータはエンキュー前にチェックされるが利用はワーカー内でのみ課金される場合、並列の玄関パスを実レースとして扱う

#### プレミアムモデル漏れ

- モデル選択、ティア分岐、プロバイダールーティングを調査
- プレミアムキーが存在するとき、無料または上限ユーザーでもプレミアムアナライザーに到達し得るか検証

#### リトライバーン

- リトライループ、重複キュージョブ、決定的失敗の再実行を調査
- 同じ非一時的エラーが分析を繰り返し消費し得るなら、品質改善の前に修正

### 5. バーン順に修正

ユーザーがコード変更を求めた場合、修正を以下の順に優先：

1. 自動 PR 重複を止める
2. クォータバイパスを止める
3. プレミアム漏れを止める
4. 重複ジョブ拡散と無意味なリトライを止める
5. 再実行/更新の安全ギャップを閉じる

同じ根本原因が明確に複数ファイルにまたがる場合を除き、パスを 1〜3 件の直接修正に絞る。

### 6. 最小の証明ステップで検証

- 変更したパスをカバーするターゲットテストまたは統合スライスのみ再実行
- バーンパスが今、以下のいずれかであるか検証：
  - ブロック
  - 重複排除
  - より安価な分析にダウングレード
  - または早期拒否
- 最終ステータスを正確に述べる：
  - ローカルで変更
  - ローカルで検証
  - プッシュ済み
  - デプロイ済み
  - まだブロック

## 高シグナルな失敗パターン

### 1. 全トリガーに単一キュータイプ

プッシュ、PR 同期、手動監査が全て同じジョブをエンキューし、ワーカーが常に PR を作るなら、分析 = PR スパム。

### 2. エンキュー後の利用予約

利用が玄関でチェックされるがワーカーでのみ増加する場合、並行リクエストが全てゲートを通りクォータを超過できる。

### 3. プレミアムパス上のフリーティア

キーが存在するとき無料のキュー済ジョブが依然として Anthropic や他のプレミアムプロバイダーにルーティングされるなら、ユーザーがプレミアム結果を見なくても実際の支出漏れ。

### 4. アプリ生成ブランチが Webhook に再突入

`pull_request.synchronize`、ブランチプッシュ、コメントトリガー実行がアプリ所有ブランチで発火すると、アプリは自分の出力を再帰的に分析できる。

### 5. 永続化安全性の前の高価な作業

システムがトークンを消費し、PR 作成、ファイル更新、ブランチ衝突で失敗し得るなら、価値を出荷せずコストを燃やしている。

## 落とし穴

- 幅広いリポジトリ徘徊から始めない。まず Webhook -> キュー -> ワーカーを落ち着かせる
- 顧客課金推論をコード裏付けのプロダクト真実と混ぜない
- 最高バーンパスが封じ込められる前に低価値の品質問題を修正しない
- 狭い証明ステップが再実行されるまでバーンが修正されたと主張しない
- ユーザーが求めない限りプッシュやデプロイしない
- 既に進行中なら無関係なリポジトリローカル変更に触れない

## 検証

- 根本原因は正確なファイルパスとコード領域を引用
- 修正はコードの美しさではなくバーンインパクト順
- 証明コマンドが名前付けされる
- 最終ステータスがローカル変更、検証、プッシュ、デプロイを区別
